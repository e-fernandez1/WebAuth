1. Project Overview

SecureAuth is a Flask web application that demonstrates production-grade authentication controls: password rules with live feedback, salted bcrypt hashing, email-based two-factor authentication, IP-based rate limiting, signed session cookies, and a real-time analytics dashboard that streams every authentication event on the system.

It is intentionally small — under 500 lines of Python — so every defense is traceable to the exact code that implements it. Each section below shows the relevant code and a one-paragraph rationale.

3. Database Schema

models.py 
class User(UserMixin, db.Model):
    id            = db.Column(db.Integer, primary_key=True)
    username      = db.Column(db.String(64), unique=True, nullable=False)
    email         = db.Column(db.String(120), unique=True, nullable=False)
    password_hash = db.Column(db.LargeBinary, nullable=False)
    created_at    = db.Column(db.DateTime, default=datetime.utcnow)


class EmailCode(db.Model):
    id         = db.Column(db.Integer, primary_key=True)
    user_id    = db.Column(db.Integer, db.ForeignKey("user.id"), nullable=False)
    code       = db.Column(db.String(6), nullable=False)
    expires_at = db.Column(db.DateTime, nullable=False)
    used       = db.Column(db.Boolean, default=False)


class AuthEvent(db.Model):
    id         = db.Column(db.Integer, primary_key=True)
    event_type = db.Column(db.String(32), nullable=False)
    username   = db.Column(db.String(64))
    detail     = db.Column(db.String(255))
    ip         = db.Column(db.String(45))
    timestamp  = db.Column(db.DateTime, default=datetime.utcnow, index=True)

Why this schema

password_hash is bytes, not text — bcrypt produces binary, and storing it as LargeBinary avoids any encoding ambiguity. EmailCode is its own table rather than a column on User because codes are short-lived, single-use objects with a clear lifecycle (issued, expires, used). AuthEvent is append-only and indexed on timestamp so the analytics dashboard can pull recent events cheaply.

4. Password Validation

app.py 
PASSWORD_RULES = {"min_length": 12, "require_upper": True, "require_lower": True,
                  "require_digit": True, "require_symbol": True}

COMMON_PASSWORDS = {"password", "password123", "qwerty123", "letmein123", ...}

def validate_password(pw):
    errors = []
    if len(pw) < PASSWORD_RULES["min_length"]:
        errors.append(f"Must be at least {PASSWORD_RULES['min_length']} characters")
    if not re.search(r"[A-Z]", pw): errors.append("Must contain an uppercase letter")
    if not re.search(r"[a-z]", pw): errors.append("Must contain a lowercase letter")
    if not re.search(r"\d",   pw): errors.append("Must contain a digit")
    if not re.search(r"[^A-Za-z0-9]", pw): errors.append("Must contain a symbol")
    if pw.lower() in COMMON_PASSWORDS: errors.append("This password is too common")
    return errors

Why 12 characters and a common-password blocklist

12 characters with mixed character classes gives ~78 bits of entropy — well past the threshold where offline cracking becomes infeasible against bcrypt at cost factor 12. NIST SP 800-63B explicitly recommends checking submitted passwords against a list of known-breached values, which is what the blocklist does. The same validate_password() runs on the server even if a client bypasses the JavaScript validator.

5. bcrypt Hashing

app.py 
# On signup: hash the password before storing
pw_hash = bcrypt.hashpw(password.encode("utf-8"), bcrypt.gensalt(rounds=12))
user = User(username=username, email=email, password_hash=pw_hash)

# On login: compare without ever decrypting (bcrypt isn't reversible)
if not bcrypt.checkpw(password.encode("utf-8"), user.password_hash):
    flash("Invalid credentials.", "danger")

Why bcrypt at cost factor 12

bcrypt is intentionally slow. Cost 12 means 2^12 = 4096 iterations of the underlying Blowfish key schedule, so a single guess takes ~200ms on commodity hardware. An attacker with a leaked database who tries one billion guesses against one user will spend over six years — assuming they have specialized hardware. The per-user salt (generated by gensalt()) defeats rainbow tables: precomputed dictionaries are useless because every hash uses a different salt.

6. Email-Based 2FA

app.py 
def issue_email_code(user):
    # Invalidate any outstanding codes for this user (single-active-code policy)
    EmailCode.query.filter_by(user_id=user.id, used=False).update({"used": True})

    # Cryptographically random 6-digit code
    code = f"{secrets.randbelow(1_000_000):06d}"
    record = EmailCode(
        user_id=user.id,
        code=code,
        expires_at=datetime.utcnow() + timedelta(minutes=5),
    )
    db.session.add(record)
    db.session.commit()
    send_verification_email(user.email, code)


def send_verification_email(to_email, code):
    """Console fallback for dev; real SMTP for production."""
    if not SMTP_HOST or not SMTP_USER or not SMTP_PASSWORD:
        print(f"[EMAIL FALLBACK]  To: {to_email}  Code: {code}")
        return True
    msg = EmailMessage()
    msg["Subject"] = "Your SecureAuth verification code"
    msg["From"] = SMTP_FROM
    msg["To"] = to_email
    msg.set_content(f"Your code is: {code}\nExpires in 5 minutes.")
    with smtplib.SMTP_SSL(SMTP_HOST, SMTP_PORT) as server:
        server.login(SMTP_USER, SMTP_PASSWORD)
        server.send_message(msg)

Why secrets.randbelow instead of random

Python's random module is deterministic given its seed and is not safe for security-sensitive randomness. secrets uses the OS cryptographic RNG (e.g. /dev/urandom on Linux, BCryptGenRandom on Windows). The single-active-code policy — invalidating prior codes when a new one is issued — closes a window where an attacker could request a code and try to use a previously-leaked one in parallel.

Why a 5-minute expiration

Long enough that a real user receives the email, switches windows, and types the code without rushing. Short enough that an intercepted code (e.g. shoulder-surfed from an email preview) is useless after a brief window. A captured code is also single-use — once verified, used = True and it cannot be replayed.

7. Verification Flow

app.py 
@app.route("/verify", methods=["GET", "POST"])
def verify():
    user_id = session.get("pending_2fa_user_id")
    if not user_id:
        return redirect(url_for("login"))
    user = db.session.get(User, user_id)

    if request.method == "POST":
        code = request.form.get("code", "").strip()
        record = (
            EmailCode.query
            .filter_by(user_id=user.id, code=code, used=False)
            .filter(EmailCode.expires_at > datetime.utcnow())
            .first()
        )
        if record:
            record.used = True
            db.session.commit()
            session.pop("pending_2fa_user_id", None)
            login_user(user)
            log_event("2fa_success", username=user.username)
            return redirect(url_for("dashboard"))
        else:
            log_event("2fa_fail", username=user.username)
            flash("Invalid or expired code.", "danger")
    return render_template("verify.html", username=user.username, email=user.email)

Why the code lookup checks four things atomically

A valid code must (1) belong to this user, (2) match the submitted digits exactly, (3) not be marked used, and (4) not be expired. All four are enforced in the same SQL query so there's no time-of-check-to-time-of-use gap. The pending user ID lives in the session, not in a form field, so an attacker who knows another user's username cannot complete their 2FA step.

8. Rate Limiting

app.py 
LOGIN_ATTEMPTS = defaultdict(list)              # ip -> list of attempt timestamps
RATE_LIMIT_WINDOW = timedelta(minutes=1)
RATE_LIMIT_MAX = 5

def is_rate_limited(key):
    now = datetime.utcnow()
    cutoff = now - RATE_LIMIT_WINDOW
    LOGIN_ATTEMPTS[key] = [t for t in LOGIN_ATTEMPTS[key] if t > cutoff]
    return len(LOGIN_ATTEMPTS[key]) >= RATE_LIMIT_MAX

def record_attempt(key):
    LOGIN_ATTEMPTS[key].append(datetime.utcnow())

# Inside /login:
if is_rate_limited(ip):
    log_event("rate_limited", username=username, detail=f"ip={ip}")
    return render_template("login.html"), 429

Why per-IP, not per-username

A username-scoped rate limit lets an attacker lock a victim out of their own account by spamming bad guesses at their username (denial of service). IP-scoped limiting forces the attacker to switch IPs — expensive for a real attack, free for a legitimate user. In production this would be backed by Redis (or fronted by a CDN like Cloudflare) so it survives restarts and works across multiple servers.

9. Session Management

app.py 
app.config["SECRET_KEY"] = os.environ.get("SECRET_KEY", "dev-secret-change-me-in-prod")

login_manager = LoginManager(app)
login_manager.login_view = "login"

@login_manager.user_loader
def load_user(user_id):
    return db.session.get(User, int(user_id))

@app.route("/dashboard")
@login_required
def dashboard():
    return render_template("dashboard.html", events=...)

Why Flask-Login + signed cookies

Flask-Login stores only the user's ID in a session cookie that is signed (not encrypted) with the app's SECRET_KEY. Tampering with the cookie invalidates the signature and the request is treated as unauthenticated. The @login_required decorator gates protected routes — attempting to hit /dashboard without a valid session redirects to login. logout_user() actively destroys the session rather than letting it linger.

10. Admin Authorization

app.py 
def admin_required(f):
    @wraps(f)
    @login_required
    def wrapper(*args, **kwargs):
        if not getattr(current_user, "is_admin", False):
            abort(403)
        return f(*args, **kwargs)
    return wrapper

@app.route("/admin")
@admin_required
def admin_home():
    users = User.query.order_by(User.created_at.desc()).all()
    events = AuthEvent.query.order_by(AuthEvent.timestamp.desc()).limit(80).all()
    return render_template("admin.html", users=users, events=events, ...)

@app.route("/admin/user/<int:user_id>/delete", methods=["POST"])
@admin_required
def admin_delete_user(user_id):
    if user_id == current_user.id:
        flash("You can't delete your own account.", "danger")
        return redirect(url_for("admin_home"))
    user = db.session.get(User, user_id)
    if user:
        EmailCode.query.filter_by(user_id=user.id).delete()
        db.session.delete(user)
        db.session.commit()
        log_event("admin_action", username=current_user.username,
                  detail=f"deleted_user={user.username}")

Why a custom decorator and self-protection

The @admin_required decorator stacks on top of @login_required, so an unauthenticated request gets redirected to login and an authenticated-but-non-admin request gets a 403. Every admin action checks user_id == current_user.id before destructive operations — this prevents an admin from accidentally locking themselves out by demoting or deleting their own account, which would otherwise leave the system with no admin. Every admin action is also logged as an admin_action event with the actor's username and the affected target, creating an audit trail.

11. Real-Time Analytics

app.py 
@app.route("/api/stats")
def api_stats():
    counts = {}
    for event_type in ["signup_success", "signup_fail", "login_fail",
                       "2fa_success", "2fa_fail", "logout",
                       "simulated_attack", "simulated_blocked"]:
        counts[event_type] = AuthEvent.query.filter_by(event_type=event_type).count()
    recent = AuthEvent.query.order_by(AuthEvent.timestamp.desc()).limit(20).all()
    return jsonify({"counts": counts, "recent": [e.to_dict() for e in recent],
                    "total_users": User.query.count()})
analytics.html 
async function refresh() {
    const res = await fetch('/api/stats');
    const data = await res.json();
    animateNumber(document.getElementById('stat-users'), data.total_users);
    // ...update bars, chart, recent feed...
}
refresh();
setInterval(refresh, 2000);   // poll every 2 seconds

Why polling instead of WebSockets

A 2-second poll is good enough for a dashboard that humans look at, and it works on any host (Render free tier, school server, anything that speaks HTTP). WebSockets or Server-Sent Events would push updates faster but add deployment complexity for a marginal UX gain. The chart shows the delta between polls — so each spike represents events that occurred in the last 2 seconds, which is what makes the brute-force simulator look dramatic.

12. Brute Force Simulator

app.py 
@app.route("/api/simulate-bruteforce", methods=["POST"])
def simulate_bruteforce():
    sim_key = f"sim-{datetime.utcnow().timestamp()}"   # isolated rate-limit bucket
    attempted, reached, blocked = 25, 0, 0
    for i in range(attempted):
        if is_rate_limited(sim_key):
            blocked += 1
            event_type = "simulated_blocked"
        else:
            record_attempt(sim_key)
            reached += 1
            event_type = "simulated_attack"
        db.session.add(AuthEvent(event_type=event_type, username=f"attacker_{i:02d}",
                                 detail="brute_force_demo", ip=request.remote_addr))
    db.session.commit()
    return jsonify({"attempted": attempted, "blocked_by_rate_limit": blocked,
                    "reached_password_check": reached, "would_have_succeeded": 0})

Why a separate sim_key per call

If the simulator shared the real per-IP rate-limit bucket, clicking the demo button would lock the user out of their own login flow for a minute. Using a fresh sim_key per simulation keeps the demo isolated. The events still show in the public analytics dashboard so the audience sees the attack pattern in real time, but real auth traffic is unaffected.

13. Why These Choices, In One Place

  • Flask, not Django: Django carries a lot of machinery for a project this size. Flask is small enough that every line of authentication logic lives in one file and is auditable.
  • SQLite, not Postgres: Single-file database, zero setup, sufficient for a demo. The SQLAlchemy abstraction means migrating to Postgres in production is a one-line config change.
  • bcrypt, not SHA256 or PBKDF2: Both are valid, but bcrypt has a tunable cost factor explicitly designed to slow down with hardware advances. Argon2 is also acceptable; bcrypt was chosen here for ubiquity.
  • Email 2FA, not TOTP-only: Universal access — anyone with an email account can use the system without installing an authenticator app. Tradeoff: email accounts can be compromised, so TOTP is technically stronger; for an educational demo, the accessibility wins.
  • In-memory rate limiter: Sufficient for a single-process demo. Production deployments behind a load balancer would use Redis to share state across workers.
  • Server-side validation duplicates the client: Client-side checks improve UX but are not security. Every rule on the signup form is re-checked in validate_password() after submission, so a user who disables JavaScript or sends a raw HTTP request gets no advantage.
  • SQLAlchemy ORM, not raw SQL: The ORM compiles every query with bound parameters, structurally preventing SQL injection. The matrix on the Security page is honest — injecting ' OR '1'='1' -- at the login form fails because the input is treated as a literal username string.
  • Live analytics: Lets the audience see defenses working in real time, not just hear about them. The dashboard, the simulator, and the threat-model page exist so the security claims are verifiable on screen during the presentation.
  • Admin role as a database column, not a hardcoded username: Storing is_admin as a User field lets administrators be promoted or demoted at runtime through the admin console — without code deploys. Self-action guards prevent the last admin from accidentally removing themselves.
