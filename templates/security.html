Defenses & Threat Model

Common attacks against authentication systems and how this application defends against each.

Transport Layer Security

Production

In production, this site is served over HTTPS with TLS 1.3. Certificates are issued by Let's Encrypt, a non-profit Certificate Authority, and chained to the ISRG Root X1 root that ships in every modern browser's trust store. Renewals happen automatically every 90 days.

Browser → Site Cert → R3 Intermediate → ISRG Root X1 → trust store.

Local Development

Locally, the dev server runs on plain HTTP because self-signed certificates trigger browser warnings that distract from the demo. The TLS layer is provided by the production host, not the application, so deploying behind any modern reverse proxy (Render, Fly.io, Cloudflare, nginx + certbot) yields HTTPS automatically.

Without TLS: passwords and TOTP codes are sent in plaintext over the wire — trivial to capture on any shared network.

Attack × Defense Matrix

Brute Force Login
Active

Attack: Try thousands of password guesses against a known username.

Defense: Per-IP rate limiter (5 attempts / minute) + bcrypt cost factor 12 makes each guess slow even if it reaches the hash check.

Credential Stuffing
Active

Attack: Replay leaked username/password pairs from other breaches.

Defense: Email-based 2FA blocks login even when the password is correct — the attacker doesn't have access to the user's email inbox.

Database Breach
Active

Attack: Steal the user table and crack credentials offline.

Defense: Passwords are stored as bcrypt hashes with per-user salts. Cost factor 12 means each guess takes ~200ms — offline brute force is days per password.

Replay Attack
Active

Attack: Capture a valid 2FA code and reuse it to log in later.

Defense: Email codes are single-use (marked used = True on first verification) and expire after 5 minutes. A captured code becomes worthless the moment it's redeemed or the window passes.

Weak Password Submission
Active

Attack: User submits "password123" expecting the system to accept it.

Defense: Server-side validation enforces length (12+), character classes, and a common-password blocklist. Client-side validation just provides feedback — the server is authoritative.

Session Hijacking
Active

Attack: Steal or forge a session cookie to impersonate a user.

Defense: Flask-Login uses cryptographically signed session cookies. Tampering invalidates the signature; logout actively destroys the session.

SQL Injection
Active

Attack: Submit ' OR '1'='1 as a username to bypass authentication.

Defense: All database queries go through SQLAlchemy's parameterized query builder — user input is never concatenated into SQL.

Eavesdropping (MITM)
Production Only

Attack: Sniff network traffic on a shared Wi-Fi to read credentials in flight.

Defense: TLS 1.3 encrypts every byte in transit. Active in production via Let's Encrypt; inactive on the local dev server.
