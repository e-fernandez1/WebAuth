Defenses & Threat Model

Every defense currently active on this site, organized by category. Each card states the attack it mitigates and how the system blocks it.

Transport Layer

Production

HTTPS via TLS 1.3. Certificates issued by Let's Encrypt and chained to ISRG Root X1 in every modern browser's trust store. Renewals automatic.

Browser → Site Cert → R3 Intermediate → ISRG Root X1.

Local Development

Plain HTTP locally so self-signed certificate warnings don't distract from the demo. TLS layer is provided by the production host (Render's edge), so deploying yields HTTPS automatically.

Without TLS: passwords sent in cleartext — trivially captured on shared networks.

Authentication & Identity

Brute Force Login
Active

Attack: Submit thousands of password guesses against a known username.

Defense: Per-IP rate limiter (5 attempts / minute) returns HTTP 429. Even attempts that pass hit bcrypt cost 12, slowing each guess to ~200ms.

Credential Stuffing
Active

Attack: Replay leaked username/password pairs from other breaches.

Defense: Email-based 2FA blocks login even when the password is correct — the attacker doesn't have access to the user's email inbox.

Database Breach
Active

Attack: Steal the user table and crack credentials offline.

Defense: bcrypt with per-user salt and cost factor 12. Each guess takes ~200ms; offline brute force is days per password. Salt defeats rainbow tables.

Replay Attack
Active

Attack: Capture a valid 2FA code and reuse it later.

Defense: Email codes are single-use (used = True on first verification) and expire after 5 minutes. A captured code is worthless after redemption or the window closes.

Weak Password Submission
Active

Attack: User submits "password123" expecting acceptance.

Defense: Server-side validation enforces 12+ characters, mixed character classes, and a common-password blocklist. Same rules re-checked even when JavaScript is disabled.

Fake Email Signup
Active

Attack: Sign up with junk like asdf@asdf to skip verification.

Defense: email-validator rejects malformed addresses and checks DNS MX records to reject domains that can't receive mail. Signup also requires confirming a code sent to the address.

Email Spoof / Unverified Account
Active

Attack: Create an account using someone else's email.

Defense: No User row is written until the signup confirmation code is verified. A pending signup expires after 10 minutes if the address holder never confirms.

Username Enumeration
Active

Attack: Probe login responses to discover which usernames exist.

Defense: All failed login attempts return the same generic "Invalid credentials" message regardless of whether the username exists or the password was wrong.

Data Protection

SQL Injection
Active

Attack: Submit ' OR '1'='1 as a username to bypass authentication.

Defense: All database queries go through SQLAlchemy ORM, which compiles to bound parameters. User input is never concatenated into SQL.

Cross-Site Scripting (XSS)
Active

Attack: Inject <script> tags via a username or other form input that gets rendered later.

Defense: Jinja2 auto-escapes all template output by default. Special characters are converted to HTML entities, so injected payloads render as inert text.

Session & Browser

Session Hijacking
Active

Attack: Steal or forge a session cookie to impersonate a user.

Defense: Flask-Login uses cryptographically signed cookies. Any tampering invalidates the signature. Logout actively destroys the session.

Cookie Theft via JavaScript
Active

Attack: A script (XSS, malicious extension) reads document.cookie and exfiltrates the session.

Defense: Session cookie is set HttpOnly — not accessible from JavaScript at all.

Cross-Site Request Forgery (CSRF)
Mitigated

Attack: Trick a logged-in user's browser into submitting a form to this site from a third-party page.

Defense: Session cookie set with SameSite=Lax — browser refuses to attach the cookie on cross-site POSTs, so a forged request hits the server unauthenticated.

Clickjacking
Active

Attack: Embed this site in a hidden iframe on a malicious page and trick users into clicking sensitive controls.

Defense: X-Frame-Options: DENY header prevents the site from being framed by any origin.

MIME-Sniff Confusion
Active

Attack: Upload a file with a misleading extension that browsers interpret as something else (e.g., HTML hosted as an image).

Defense: X-Content-Type-Options: nosniff forces browsers to honor the declared Content-Type.

Eavesdropping (MITM)
Production Only

Attack: Sniff network traffic on shared Wi-Fi to read credentials in flight.

Defense: TLS 1.3 encrypts every byte in transit. Strict-Transport-Security header tells browsers to require HTTPS for the next year.

Authorization & Audit

Privilege Escalation
Active

Attack: A regular user calls an admin endpoint directly via crafted HTTP.

Defense: Custom @admin_required decorator wraps every admin route. Non-admins get HTTP 403.

Admin Self-Lockout
Active

Attack: An admin accidentally demotes or deletes themselves, leaving no admin in the system.

Defense: Self-action guards in admin routes refuse to act when user_id == current_user.id.

Insufficient Audit Trail
Active

Attack: An attacker takes admin actions and the log doesn't capture who did what.

Defense: Every signup, login, 2FA, logout, rate-limit, and admin action is written to the AuthEvent table with username, IP, timestamp, and detail. Streamed live to /analytics.

Information Leak via Referrer
Active

Attack: A user clicks an external link; the destination sees the full URL of where they came from.

Defense: Referrer-Policy: strict-origin-when-cross-origin only sends the origin (no path or query) to cross-site navigations.

HTTP Security Headers Set on Every Response

HeaderValueWhat it does
X-Content-Type-Options nosniff Disables browser MIME-type sniffing
X-Frame-Options DENY Blocks the site from being embedded in any iframe (clickjacking)
Referrer-Policy strict-origin-when-cross-origin Only leaks the origin (not the path) to other sites
Permissions-Policy geolocation=(), microphone=(), camera=() Disables sensitive browser APIs the site doesn't use
Strict-Transport-Security max-age=31536000; includeSubDomains Forces HTTPS for one year (only sent over HTTPS)
Set-Cookie HttpOnly; SameSite=Lax; Secure (prod) Session cookie is unreadable by JS, blocks cross-site sends, HTTPS-only in production
